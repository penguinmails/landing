# Documentation Writing Style Guide

This guide defines the expected tone and structure for documentation in this repository, including docs generated by AI agents.

When `AGENTS.md` instructions lead to new documentation, that documentation should follow this guide.

## Audience-First Structure

Start with non-technical framing, then move progressively into implementation detail.

- First section: product context and user/business impact in plain language.
- Middle section: behavior, workflows, and decisions with light technical detail.
- Final section: technical details, implementation notes, and code references.

If a section becomes too technical for the main narrative, split it into a separate deep-dive document and link to it from the parent doc.

## Core Writing Principles

Write direct statements with references embedded naturally in prose. Avoid stacking disconnected bullet lists when a short paragraph communicates intent better.

Keep each paragraph focused on one idea:

1. Main point.
2. Supporting detail.
3. Concrete example.
4. Link to code, decision, or deep-dive doc when relevant.

Use active, specific language. Prefer "what happens" over "what could happen." Remove filler transitions and generic AI-like phrasing.

## Tone and Readability

Use concise prose that is easy for product, design, and engineering readers to follow.

- Prefer short paragraphs over large blocks.
- Use bullets only when list structure clearly improves readability.
- Avoid excessive bullet points for narrative sections.
- Keep terminology consistent across docs (`landing`, `dashboard`, `CTA`, `intent`, `handoff`).

## Reference and Linking Rules

Embed references in sentence flow rather than adding isolated "see X" callouts at the end.

Good pattern:

```markdown
The landing site captures plan intent and forwards users to the dashboard, where checkout is executed (`src/lib/cta.ts`; `DECISIONS.md`).
```

Use links intentionally:

- Link from non-technical docs to deeper technical docs.
- Link behavior claims to code when practical.
- Link stable product rationale to `DECISIONS.md`.

## Non-Technical to Technical Progression Template

Use this structure for most substantial docs:

```markdown
# Title

## What This Means
[Plain-language context, scope, and why this matters.]

## How It Works
[Behavior flow, ownership boundaries, and expected outcomes.]

## Implementation Notes
[Technical details, key files/components, data flow, constraints.]

## Deep Dives
- [Specific technical deep dive](./related-deep-dive.md)
```

## Deep-Dive Split Guidance

Create a separate deep-dive doc when:

- detailed code walkthroughs interrupt narrative readability,
- low-level implementation options are extensive,
- multiple edge cases need dense technical treatment.

Main docs should summarize decisions and outcomes; deep dives should carry full technical analysis.

## Preferred Patterns

### Pattern: Direct assertion with embedded support

```markdown
Landing pages collect intent while billing executes in the dashboard (`DECISIONS.md`; `src/lib/cta.ts`).
```

### Pattern: Paragraph with one focus

```markdown
Pricing CTAs communicate plan selection and attribution context. They preserve `plan`, `billing`, and UTM parameters during handoff to the dashboard so downstream flows can personalize onboarding (`src/lib/cta.ts`).
```

### Pattern: Narrative first, details linked

```markdown
Users start on marketing pages and complete account actions in the product dashboard. Technical implementation details are documented in the pricing handoff deep dive (`docs/pricing-handoff-deep-dive.md`).
```

## Anti-Patterns to Avoid

- Opening directly with low-level implementation details before context.
- Long bullet-only sections where prose would be clearer.
- Generic transitions like "This means..." and "As a result of this...".
- Hypothetical phrasing without concrete behavior.
- Standalone reference callouts detached from paragraph flow.
- Mixing product explanation and deep technical internals in the same section when split docs would read better.

## Documentation Review Checklist

- [ ] Starts with non-technical context before technical detail.
- [ ] Uses short prose paragraphs with one clear idea each.
- [ ] Includes concrete examples, not abstract claims.
- [ ] Embeds links/references naturally in sentence flow.
- [ ] Avoids excessive bullet points in narrative sections.
- [ ] Splits deep technical detail into dedicated docs when needed.
- [ ] Links decisions to `DECISIONS.md` where relevant.
- [ ] Matches repository vocabulary and behavior boundaries.
